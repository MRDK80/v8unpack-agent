# Структура распакованных форм: discovery по *.elem.json

Подтверждено на реальных и синтетических данных: issue #55 (июль 2026).

v8unpack 1.2.11 материализует каждую форму (обычную и управляемую) как
`*.elem.json` в каталоге формы. `Form.xml` в pipeline v8unpack-agent не
используется.

> **Примечание.** По итогам проверки на 2216 формах реальной конфигурации
> установлено, что `*.elem.json` генерируется для **всех** форм — обычных
> и управляемых. Классификация ordinary/managed не входит в scope этого
> discovery — см. issue #56.

## Layout-варианты путей

Discovery устойчив к глубине: контейнер определяется по суффиксу `Form`
родительского каталога формы, независимо от числа уровней.

### 4-уровневый layout (Catalog, Document и др.)

    <root>/<object_type>/<object_name>/<ContainerForm>/<form_name>/
    ├── <form_name>.elem.json           # структура формы (обязательный)
    ├── <form_name>.<descent>.json      # метаданные формы v8unpack (опциональный)
    └── <form_name>.obj.<descent>.bsl   # модуль формы (опциональный)

Примеры:

    Catalog/Банки/CatalogForm/ФормаЭлементаУправляемая/CatalogForm.elem.json
    Document/АвансовыйОтчет/DocumentForm/ФормаДокументаУправляемая/DocumentForm.elem.json

### 3-уровневый layout (CommonForm)

    <root>/CommonForm/<form_name>/
    └── CommonForm.elem.json

Пример:

    CommonForm/ФормаВыбораПериода/CommonForm.elem.json

### 3-уровневый layout (внешние объекты — EPF / ERF)

    <root>/<object_name>/Form/<form_name>/
    └── Form.elem.json

    <root>/<object_name>/ReportForm/<form_name>/
    └── ReportForm.elem.json

Примеры:

    ВнешняяОбработкаУпр/Form/ФормаВнешняяУправляемая/Form.elem.json
    ВнешнийОтчетУправляемый/ReportForm/ФормаОтчетаУправляемая/ReportForm.elem.json

## Поддерживаемые контейнеры форм

| Суффикс каталога | Назначение |
|------------------|------------|
| `CatalogForm`    | Формы справочников |
| `DocumentForm`   | Формы документов |
| `Form`           | Формы обработок / внешних обработок |
| `ReportForm`     | Формы отчётов / внешних отчётов |
| `CommonForm`     | Общие формы конфигурации |

Поддерживается любой каталог с суффиксом `Form` (например `TaskForm`,
`BusinessProcessForm` и т.д.).

## Определение «форма с элементным представлением»

Форма обнаруживается, если в её каталоге присутствует хотя бы один файл
`*.elem.json`. Тип формы (ordinary/managed) не определяется — см. issue #56.

## Артефакты каталога формы

Реальная схема v8unpack 1.2.11 (распаковка без `--descent`):

| Артефакт | Обязательный | Поле ElemFormEntry | Описание |
|----------|--------------|--------------------|----------|
| `<stem>.elem.json` | ✅ | `elem_json_path` | Структура формы |
| `<stem>.json`      | ❌ | `meta_json_path`  | Метаданные формы |
| `<stem>.id.json`   | ❌ | `id_json_path`    | UUID формы (`{"uuid": ...}`) |
| `<stem>.obj.bsl`   | ❌ | `bsl_path`        | BSL-модуль формы |

Модули **не** форм игнорируются: `<stem>.mgr.bsl` (менеджер объекта),
`Configuration.seance/.con/.app/.802.bsl` (модули конфигурации). Модуль формы —
строго `<stem>.obj.bsl`.

> **Важно.** Суффикс `id` в `<stem>.id.json` — это отдельный UUID-файл, а не
> значение `--descent`. Он всегда попадает в `id_json_path` и никогда не
> трактуется как descent-набор.

## Descent-суффикс (распаковка с `--descent`)

Если пользователь распаковал конфигурацию с числовым `--descent`, сопутствующие
файлы получают суффикс и группируются в `descent_artifacts`:

| `--descent`   | вспомогательный JSON       | BSL-модуль                    |
|---------------|----------------------------|-------------------------------|
| `10`          | `<stem>.10.json`           | `<stem>.obj.10.bsl`           |
| `3.0.75.100`  | `<stem>.3.0.75.100.json`   | `<stem>.obj.3.0.75.100.bsl`   |

Правила:

- Значение descent — число из 1–4 компонентов по 1–3 цифры (литерал `id` — не descent).
- JSON и BSL объединяются в один `DescentArtifacts` при совпадении суффикса.
- При нескольких descent-наборах возвращаются все, добавляется предупреждение в `extra_warnings`.
- На типовой распаковке (без `--descent`) `descent_artifacts` пуст.

Подтверждено на реальной конфигурации (2212 форм, июль 2026): все формы имеют
`<stem>.json` и `<stem>.id.json`, 2164 — `<stem>.obj.bsl`, descent-суффиксов нет.

### Наблюдение по реальным данным: basename файла

На синтетических фикстурах файл структуры назывался по имени формы —
`<form_name>.elem.json`. На реальных распаковках v8unpack 1.2.11 это не всегда
так: файл часто называется по имени контейнера, например `CatalogForm.elem.json`,
`DocumentForm.elem.json`, `Form.elem.json`, `ReportForm.elem.json`,
`CommonForm.elem.json`.

Discovery не зависит от точного basename файла: форма определяется по наличию
любого `*.elem.json` в каталоге формы, поэтому корректно обрабатываются оба
варианта именования.

## Гипотеза классификации ordinary/managed (задел для issue #56)

Проверка на 2216 формах реальной конфигурации (v8unpack 1.2.11) выявила
эмпирический признак в `*.elem.json`:

- `props[0].raw[0] == "9"` → предположительно **управляемая** форма (857 шт.)
- `props[0].raw[0]` — список (напр. `["0"]`) → предположительно **обычная** форма (1076 шт.)
- `props` пустой / отсутствует → не определено (283 шт.)

> **Важно.** Это результат реверс-инжиниринга бинарного формата 1С без
> официальной спецификации. Признак хрупкий — полагаться на него в
> production-коде нельзя без дополнительной верификации. Надёжная
> реализация классификатора — отдельный issue #56.

### Верификация: эксперимент со страницами (июль 2026)

Для проверки устойчивости признаков в управляемую форму
`Catalog/Банки/CatalogForm/ФормаЭлементаУправляемая` были добавлены страницы
через конфигуратор 1С, после чего конфигурация перераспакована (`v8unpack -E`).

Сводка наблюдений (три точки):

| Признак | Обычная форма | Управляемая (до страниц) | Управляемая (после страниц) |
|---------|---------------|--------------------------|------------------------------|
| `props[0].raw[0]` | список `['0']` | строка `'9'` | **строка `'9'`** ✅ |
| `params` | `[]` (пустой список) | `null` | **`null`** ✅ |
| `page` у узлов tree | 18/18 | 0/13 | **0/13** |
| `-pages-` в `data` | `True` | `False` | **`False`** |
| ключи с `/` в `data` | 18 | 0 | **5** (новые страницы, но другая кодировка) |

**Выводы:**

1. **`props[0].raw[0]` — основной признак** (строка `"9"` → управляемая).
   Не изменился после добавления страниц: устойчив к изменению layout контента.
2. **`params == null` — подтверждающий признак** (у обычной — `[]`).
   Тоже устоял.
3. **`page` / `-pages-` — отвергнуты как критерий типа формы.**
   Даже после добавления страниц у управляемой формы `page`/`-pages-` не
   появились. Страницы кодируются иначе (ключи с `/` в `data` без маркера
   `-pages-`). Эти поля отражают **способ кодирования layout** в обычных
   формах, а не сам факт «есть страницы» или «это обычная форма».

Рекомендуемый признак для реализации #56: **`props[0].raw[0]`** (основной)
+ **`params`** (подтверждающий). Признаки `page`/`-pages-` не использовать.

## Отличия носителей: *.elem.json у обычных vs управляемых форм

Top-level структура `*.elem.json` одинакова у обеих (`params/props/commands/tree/data`).
Различия наблюдаются на уровне содержимого:

| Аспект | Обычная форма | Управляемая форма |
|--------|---------------|-------------------|
| `props[0].raw[0]` | список (напр. `['0']`) | строка `"9"` |
| `params` | `[]` пустой список | `null` |
| `data` | иерархия страниц/элементов (`Страница1/Код`, `-pages-`) | другая раскладка (без `-pages-`; плоский словарь или иерархия с `/` но без `-pages-`) |

## Маппинг в ElemFormEntry

- `elem_json_path` → `<stem>.elem.json` (обязательный, относительный)
- `meta_json_path` → `<stem>.json` или `None`
- `id_json_path`   → `<stem>.id.json` или `None`
- `bsl_path`       → `<stem>.obj.bsl` (или первый descent-BSL) или `None`
- `descent_artifacts` → наборы для числовых `--descent` (пусто на типовой распаковке)
- `extra_warnings` → нефатальные предупреждения

Обратная совместимость (deprecated):

- `aux_json_path` — property, возвращает `meta_json_path` (или первый
  descent JSON). Используйте `meta_json_path`.

## Использование

```python
from pathlib import Path
from v8unpack_agent.managed_forms import discover_elem_forms

forms = discover_elem_forms(Path("path/to/unpacked/cf"))
for form in forms:
    print(form.elem_json_path)
```

## Связь с реестром форм (issue #57)

`scan_forms` использует `discover_elem_forms` для подбора форм **без кода**
(без `.obj.bsl`) в единый `FormScanIndex`: `ElemFormEntry.elem_json_path`
переносится в `FormEntry.elem_json_path` (тот же относительный путь), а
недостающие метаданные (`object_type` / `object_name` / `container_name` /
`form_name`) восстанавливаются из пути формы с учётом режима сканирования
(config / external). Подробности — [docs/scan_forms.md](scan_forms.md),
раздел «Формы без кода (elem-only, issue #57)».

## Связь с выжимкой формы (issue #66)

Каноническую структуру формы из `*.elem.json` даёт `parse_elem_json`, а
семантическую выжимку поверх неё строит `build_managed_form_summary(form_dir)`
(issue #66, PR #68). Отдельного адаптера реального формата нет:
`parse_elem_json` — единственный парсер `*.elem.json`. Подробности —
[docs/managed_form_summary.md](managed_form_summary.md).
